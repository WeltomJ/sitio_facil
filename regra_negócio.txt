# 🏛️ 1. A Essência do Sistema (Visão Estratégica)

Você está criando um **marketplace de aluguel de chácaras**, com dois agentes principais:

* **Locador** (oferta)
* **Cliente** (demanda)

E um ativo central:

* **Chácara (imóvel)**

O sistema precisa resolver 3 problemas:

1. **Descoberta** (buscar locais)
2. **Disponibilidade** (datas/horários)
3. **Reserva** (com controle de conflito)

---

# ⚖️ 2. Regras de Negócio Fundamentais

## 👤 2.1 Cadastro de Usuários

**Regra 1:** Todo usuário deve se cadastrar como:

* Pessoa Física (CPF)
* Pessoa Jurídica (CNPJ)

**Regra 2:** Um usuário pode ser:

* Cliente
* Locador
* Ou ambos

**Regra 3:** Não pode haver duplicidade de CPF/CNPJ no sistema.

---

## 🏡 2.2 Cadastro de Chácaras

**Regra 4:** Apenas usuários com perfil de locador podem cadastrar chácaras.

**Regra 5:** Cada chácara deve conter:

* Nome
* Endereço (com integração com mapa)
* Descrição
* Capacidade máxima
* Preço (por diária ou por período)
* Lista de comodidades (ex: piscina, churrasqueira)

**Regra 6:** Cada chácara pertence a apenas um locador.

---

## 🕒 2.3 Disponibilidade e Agenda

Aqui está o coração do sistema — se errar isso, tudo quebra.

**Regra 7:** O locador define:

* Horário de check-in
* Horário de check-out

**Regra 8:** Uma reserva ocupa um intervalo de tempo:

```
data_início + hora_checkin → data_fim + hora_checkout
```

**Regra 9:** Não pode haver sobreposição de reservas confirmadas.

**Regra 10:** Reservas pendentes NÃO bloqueiam a data.

(essa foi sua ideia — e é correta para simplicidade)

---

## 🔍 2.4 Busca de Chácaras

**Regra 11:** O cliente pode buscar por:

* Localização (via mapa ou raio)
* Datas desejadas
* Capacidade mínima

**Regra 12:** O sistema deve mostrar apenas chácaras disponíveis no período.

---

## 📅 2.5 Reserva

**Regra 13:** Ao solicitar reserva:

* Status inicial = **PENDENTE**

**Regra 14:** Uma reserva só se torna válida quando:

* Pagamento é confirmado (mesmo que manual)

**Regra 15:** Enquanto estiver pendente:

* Outros usuários podem reservar o mesmo período

**Regra 16:** Quando confirmada:

* O sistema bloqueia o período

**Regra 17:** Caso duas reservas concorram:

* A primeira confirmada vence

(essa regra é importante — demonstra lógica de concorrência)

---

## 💰 2.6 Pagamento (Simplificado)

**Regra 18:** O sistema não precisa processar pagamento real.

Pode usar:

* "Pagamento simulado"
* Ou "Confirmação manual pelo locador"

**Regra 19:** O locador pode:

* Aprovar
* Recusar reservas

---

## 📜 2.7 Registro e Histórico

**Regra 20:** Todas as ações devem ser registradas:

* Criação de reserva
* Confirmação
* Cancelamento

**Regra 21:** O usuário pode visualizar:

* Histórico de reservas

# 🧠 4. Regras Inteligentes (Diferencial)

Aqui está o que separa um projeto comum de um bom projeto:

### 🔒 Controle de concorrência

Evitar que duas pessoas confirmem ao mesmo tempo:

* Pode usar validação no backend

---

### 📍 Integração com mapas

Você pode usar:

* Google Maps API
* Ou OpenStreetMap (gratuito)

Função:

* Mostrar localização
* Buscar por proximidade

---

### ⭐ Avaliações (opcional, mas forte)

**Regra extra:**

* Cliente pode avaliar após uso

---

### 📸 Fotos da chácara

* Dá valor visual ao app

---

### 🔔 Notificações simples

* Reserva confirmada
* Reserva recusada

---

# ⚔️ 5. Problemas que você PRECISA antecipar

Se você falar disso na apresentação, ganha respeito:

* Conflito de reservas
* Datas inválidas
* Usuário tentando reservar no passado
* Locador não respondendo reserva

---

# 🧭 6. Possível Fluxo do Sistema

**Cliente:**

1. Cadastro
2. Busca chácara
3. Escolhe datas
4. Faz reserva (pendente)
5. Aguarda confirmação

**Locador:**

1. Cadastro
2. Cadastra chácara
3. Recebe pedido
4. Confirma ou recusa
